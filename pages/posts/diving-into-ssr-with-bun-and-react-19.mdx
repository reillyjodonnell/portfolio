But there‚Äôs a fundamental flaw in this idea.

Anything that crosses network boundaries has to be serialized - fancy word for a string.

So how do we turn Hello and Bye to strings?

if we invoke via `<Hello />` That's not getting the underlying string bc that goes through React and instantiates a fiber and all that. What we're really wanting / needing is the underlying HTML that represents this code.

React offers 2 api‚Äôs to get the underlying HTML and it‚Äôs the basis of all SSR apps. `renderToString` which transforms your react tree into html and `renderToReadableStream` which does the same but streams it to better scale / gets you a faster Time To First Byte TTFB.

We‚Äôll do the stream approach but both work :D

Make sure you‚Äôve got react-dom which is where the above api comes from

`bun i react-dom`

## Streams in bun

Streams are as simple as passing it to the Response as the first argument:

```typescript
import React from 'react';
import { renderToReadableStream } from 'react-dom/server';

Bun.serve({
  port: 3000,
  routes: {
    '/': async () => new Response(await renderToReadableStream(<Hello />)),
    '/bye': async () => new Response(await renderToReadableStream(<Bye />)),
  },
});

console.log('Server running on http://localhost:3000');

function Hello() {
  const [name, _] = React.useState('World');
  return <h1>Hello, {name}!</h1>;
}

function Bye() {
  return <h1>Goodbye!</h1>;
}
```

When you run the server and visit the above routes it‚Äôs working!

### Quick aside: Important React ideas before we go further

React needs an entry point in the HTML. It‚Äôs a major part of HOW react works under the hood - specifically it‚Äôs diffing algorithm.

Which means we‚Äôll need to give react an entry point - canonically this is some html `id` we target.

Then we need to ‚Äúattach‚Äù react to that targeted entry point via a `hydrateRoot` call

to implement we‚Äôll need the html created from `renderToReadableStream` then a js file containing a `hydrateRoot` call to attach react.

Back to the code:

## Providing an entry point for React

From the docs for one of the arguments to `renderToReadableStream`

> - **optional** `bootstrapScripts`: An array of string URLs for the `<script>` tags to emit on the page. Use this to include the `<script>` that calls [`hydrateRoot`.](https://react.dev/reference/react-dom/client/hydrateRoot) [Omit it if](https://react.dev/reference/react-dom/client/hydrateRoot) you don‚Äôt want to run React on the client at all.

So we‚Äôll need to prep a js file and attach it as the arg.

The gist of what the script needs to do is -

1. target some html node
2. pass that node into `hydrateRoot` as well as the react node we want to render (Hello/ Bye) - we‚Äôll need to import

\[here‚Äôs how it‚Äôs done in the docs\] ([https://react.dev/reference/react-dom/client/hydrateRoot#:~:text=import%20%7B%20hydrateRoot,reactNode)%3B](<https://react.dev/reference/react-dom/client/hydrateRoot#:~:text=import%20%7B%20hydrateRoot,reactNode)%3B>))

```javascript
import { hydrateRoot } from 'react-dom/client';
import { Hello } from '.';
const domNode = document.getElementById('root');
if (domNode) hydrateRoot(domNode, <Hello />);
```

Then we‚Äôll need to include this file (main.js) in the arguments passed like so:

```typescript
new Response(
        await renderToReadableStream(
          `
      <html>
        <head>

        <div id="root">

        </div>
        </body>

          {
            bootstrapScripts: ['/main.js'],
```

This doesn‚Äôt work though!

To understand let‚Äôs go look at what `renderToReadableStream` \[here\]([https://react.dev/reference/react-dom/server/renderToReadableStream#parameters](https://react.dev/reference/react-dom/server/renderToReadableStream#parameters))

The first parameter is:

> A React node you want to render to HTML. For example, a JSX element like `<App />`. It is expected to represent the entire document, so the `App` component should render the `<html>` tag.

So we‚Äôll need to update and have it render html instead of a string.

Then for the second object it says:

> - **optional** `bootstrapScripts`: An array of string URLs for the `<script>` tags to emit on the page. Use this to include the `<script>` that calls [`hydrateRoot`.](https://react.dev/reference/react-dom/client/hydrateRoot) [Omit it if](https://react.dev/reference/react-dom/client/hydrateRoot) you don‚Äôt want to run React on the client at all.

Further down it says:

`React will inject the doctype and your bootstrap <script> tags into the resulting HTML stream:`

Let‚Äôs make the first change and take a look at the response from the server.

We get a 404 for main.js!

The request that comes in for /main.js looks something like `localhost:3000/main.js` since there‚Äôs not a route covered / we‚Äôre not handling a fallback it automatically returns a 404.

Also remember earlier when we talked about renderToString (which is blocking in that the response isn‚Äôt send until the entire react tree is rendered and turned into a string?) and it‚Äôs alt the renderToReadableStream? There‚Äôs the two same ideas for reading a file and returning it‚Äôs contents for http servers. Thankfully Bun makes it REALLY easy to stream a file. \[docs here\]([https://bun.sh/docs/api/http#streaming-files](https://bun.sh/docs/api/http#streaming-files))

Their example is

```typescript
Bun.serve({
  fetch(req) {
    return new Response(Bun.file('./hello.txt'));
  },
});
```

That‚Äôs dead simple. Under the hood Bun is auto streaming it so no matter how big the file is it will always respond quickly!

Let‚Äôs put these 2 ideas - handling the request for /main.js and serving the contents of a file as a stream together:

```typescript
import React from 'react';
import { renderToReadableStream } from 'react-dom/server';

Bun.serve({
  port: 3000,
  routes: {
    '/': async () => {
      return new Response(
        await renderToReadableStream(<Hello />, {
          bootstrapScripts: ['/main.js'],
        })
      );
    },

    '/bye': async () => new Response(await renderToReadableStream(<Bye />), {}),
  },
  fetch(req) {
    if (req.url === 'main.js') new Response(Bun.file('main.js'));
    return new Response('Not Found', { status: 404 });
  },
});

console.log('Server running on http://localhost:3000');

      </body>

  );
}

export function Bye() {

}
```

### hydrateRoot / wtf even is hydration

In trying to debug this cpu issue I found some gaps in my understanding.

To start the definition is really confusing imo:

> `hydrateRoot` lets you display React components inside a browser DOM node whose HTML content was previously generated by [`react-dom/server`](https://react.dev/reference/react-dom/server)

Further down in the docs on hydrateRoot there‚Äôs a bit more helpful info:

> Call hydrateRoot to ‚Äúattach‚Äù React to existing HTML that was already rendered by React in a server environment‚Ä¶ hydrateRoot() expects the rendered content to be identical with the server-rendered content.

that explains everything we need to know: when you render html from react, the rendered content has to be identical to the server-content and since hydrate root expects an entry - it has to be THE entry, i.e. the html tag aka document.

```typescript
  entrypoints: ['./main.tsx'],
  outdir: 'dist',
});

Bun.serve({
  port: 3000,
  routes: {
    '/': async () => {
      try {
        const stream = await renderToReadableStream(<Hello />, {
          bootstrapModules: ['main.js'],

        return new Response(stream, {
          headers: {
            'Content-Type': 'text/html',
          },
          status: 200,
        });
      } catch (error) {
        console.error('Error rendering Hello component:', error);
        return new Response('Internal Server Error', { status: 500 });
      }

    // '/bye': async () => new Response(await renderToReadableStream(<Bye />), {}),
  },
  async fetch(req) {
    const path = new URL(req.url).pathname;
    if (path === '/main.js')

          'Content-Type': 'application/javascript',
```
